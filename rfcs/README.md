# DevPulse RFCs — KPI Methodology Proposals

DevPulse measures engineering health through five KPIs. The methodology behind each one — what to measure, how to measure it, where to draw thresholds — involves real trade-offs that deserve open debate.

These RFCs are **living proposals**, not settled specifications. They represent our best thinking so far. We want your help improving them.

## What Is an RFC?

Each RFC documents a single KPI: how it's currently defined, why we chose that approach, and the open questions we're still wrestling with. They're not implementation specs — they're invitations to think through methodology together.

## How to Participate

**Comment on an RFC.** Open a pull request with suggested changes to any RFC document. Explain your reasoning — we value context over opinions.

**Start a Discussion.** For broader questions that don't map to a specific edit, open a thread in [GitHub Discussions](https://github.com/scalefirstai/DevPulse/discussions) and tag it with `methodology`.

**Share real-world data.** The most valuable contributions come from teams who've tried measuring these things. If your experience confirms or contradicts our assumptions, we want to hear about it.

## RFC Lifecycle

- **Draft** — Open for discussion. The methodology may change based on community input.
- **Accepted** — Community consensus reached. The definition is stable but can still evolve through a new RFC.
- **Superseded** — Replaced by a newer RFC.

All five RFCs are currently in **Draft** status. We expect them to evolve significantly based on community input.
